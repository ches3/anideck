# プロジェクトガイドライン

## プロジェクト概要

- ローカルに保存されたアニメをブラウザで再生するアプリケーション
- ローカルサーバー + Webフロントエンドの構成
- 対象の動画ファイルは、ユーザーが正規表現ルールにより指定する

## 開発時の品質チェック

- **コード変更時やコミット前には必ず以下の2つのコマンドを実行してエラーを修正する**
- ユーザーに実行させるのではなく、コーディングエージェント自身でこれを実行する

```bash
vp check --fix
vp test
```

## コンポーネント設計

- 1コンポーネント1責務の原則を守る
- 複雑なロジックはカスタムフックまたはユーティリティ関数に分離

## スタイリング

- Tailwind CSS v4を使用
- 基本的なUIはshadcn/uiを使う
- shadcn/uiコンポーネントの追加はコマンドで行う
- 条件付きのクラス名結合には、テンプレートリテラルではなく`cn()`を使う
- グレー系の色は`gray`ではなく`neutral`を使う
- モダンでシンプルなデザインを心がける

## テスト

- **すべての仕様をテストケースに落とし込む**
- **詳細な実装に依存せず、機能・要件を検証するためのテストにする**
- コード変更時(機能実装・バグ修正など)には、テストの追加・修正が必要かを確認する
- 実装の前にテストを追加・修正する

## パッケージ間インポートのルール

- パッケージ間のインポートは必ずパッケージ名（`@anideck/パッケージ名`）を使う
- 相対パス（`../../other-package/src/...`）でのクロスパッケージインポートは禁止

## DB

- マイグレーションファイルは `bun run db:generate` で生成する
- schemaで定義できないマイグレーションが必要な場合のみ、マイグレーションファイルを編集する
- **マイグレーションファイルの直接編集は最終的な手段であり、基本的にはコマンドで生成をするべきである**
- コミット済みのマイグレーションファイルの編集は禁止
- schemaの変更を伴わないマイグレーションが必要な場合は、`bunx drizzle-kit generate --custom` で空のマイグレーションファイルを生成して編集する
